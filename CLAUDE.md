## Project Overview

F1 GOAT Determiner determines the greatest F1 driver of all time using clustering analysis on historical race data. It is a polyglot monorepo with three services and a reverse proxy:

- **data-sync-svc** (Kotlin/Spring Boot): Fetches F1 data from Jolpica API, stores in PostgreSQL.
- **analytics-api** (Python/FastAPI): Reads from PostgreSQL, performs clustering, serves REST API.
- **frontend** (React/TypeScript/Vite): Displays tier list results.
- **nginx**: Reverse proxy serving the frontend static files and routing API calls.

Data flow: Jolpica F1 API → data-sync-svc → PostgreSQL → analytics-api → frontend

## Architecture

### data-sync-svc Module Structure

Hexagonal architecture with strict dependency direction:

```
app → jolpica → use-case → domain
app → db → domain
app → scheduled
```

- **domain**: Pure business entities and types. No framework dependencies.
- **use-case**: Sync orchestration and entity syncers. Defines port interfaces (e.g., `F1DriverFetcher`, `F1RaceResultFetcher`) that infrastructure implements.
- **jolpica**: Implements the fetcher port interfaces. Calls Jolpica REST API with rate limiting (Resilience4j) and retry.
- **db**: Spring Data JPA/JDBC repositories, Flyway migrations in `db/src/main/resources/db/migration/`.
- **scheduled**: ShedLock-based cron scheduling (`SyncScheduler`).
- **app**: Spring Boot entry point, wires all modules.
- **build-logic**: Shared Gradle convention plugins.

Sync jobs are tracked in the DB (`sync_jobs`, `sync_checkpoints` tables). Three sync commands: `FullSyncCommand`, `IncrementalSyncCommand`, `ResumeSyncCommand`.

### analytics-api Structure

- `src/main.py`: FastAPI app and (currently) most route handlers. This reflects the legacy layout; per the Python style guide, **do not add new routes here**. Existing routes will be migrated out over time.
- `src/analyzers/`: Analysis logic using Pandas/Scikit-learn (clustering).
- `src/api/`: Target location for FastAPI `APIRouter` modules and route handlers. Currently contains `__init__.py` and any newly added routers; new routes should be defined here rather than in `src/main.py`.
- `src/core/`: Config (`pydantic-settings`), data loading, preprocessing.
- `src/models/schemas.py`: Pydantic v2 response schemas.

Currently reads from CSV files in `data/raw/`; being migrated to query PostgreSQL.

### frontend Structure

- `src/pages/`: Top-level page components (`HomePage`, `TierListPage`).
- `src/components/`: UI components grouped by domain (`common/`, `layout/`, `tierList/`).
- `src/hooks/`: Custom React hooks (e.g., `useTierList.ts`).
- `src/api/`: API client (`client.ts`) and typed request functions (`tierList.ts`, `types.ts`).
- `src/utils/`: Shared utilities (e.g., `pluralize.ts`).
- `src/styles/`: Global SCSS styles and variables.

### Database

PostgreSQL on port 5433 locally. Managed via Flyway (migrations in `data-sync-svc/db/src/main/resources/db/migration/`).

Tables: `statuses`, `circuits`, `constructors`, `drivers`, `races`, `results`, `qualifying`, `driver_standings`, `constructor_standings`, `sync_jobs`, `sync_checkpoints`, `shedlock`.

Default local credentials: host `localhost:5433`, db `f1_goat_determiner`, user `f1user`, password `f1password`.

## data-sync-svc Coding Rules

- **Logging**: Use `KotlinLogging.logger {}` from `io.github.oshai:kotlin-logging-jvm`. Never use `LoggerFactory` directly.
- **IDs**: Tables use `GENERATED BY DEFAULT AS IDENTITY`. Domain models use `val id: Int? = null`. Never generate IDs manually. Use natural keys (`ref`, `(season, round)`, `(race_id, driver_id)`) for upsert conflict resolution.
- **Exception handling**: Never `try-catch` with `Exception`/`RuntimeException`/`Throwable` (detekt forbids it). Use `runCatching { ... }.fold(...)` or `.onFailure { ... }` for fault-tolerant code.
- **Detekt**: Fix violations in code; never modify `config/detekt/detekt.yml` to suppress them.
- **Kotlin idioms**: Prefer `val` with `get()` over no-arg functions when the member has no parameters and returns a value (e.g., `val hamilton get() = Driver(...)` instead of `fun hamilton() = Driver(...)`). This applies to test fixture factories, builders, and similar patterns. Use functions only when parameters are required. Prefer Kotlin extensions and reified overloads whenever they exist. Use `context.getBean<MyType>()` over `context.getBean(MyType::class.java)`, `jdbcTemplate.queryForObject<Long>(...)` over `jdbcTemplate.queryForObject(..., Long::class.java)`, and the same rule for similar APIs like `readValue<T>()` / `body<T>()`. Fall back to `Class<T>`/`KClass<T>` only when there is no Kotlin extension.
- **Testing**: Kotlin tests must use Kotest assertions only, including `shouldThrow<T>()` for exception checks. Integration and e2e tests must be independent, runnable in any order, and must fully reset or recreate their own state. Keep assertions strict and exact; if a strict check fails because of shared state, fix test isolation instead of weakening the assertion.
- **Dependency inversion**: `use-case` must never import from `jolpica` or `db`. New external API calls require a port interface in `use-case` implemented by `jolpica`.
